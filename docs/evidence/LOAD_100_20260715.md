# 100개 실제 JPEG worker 부하 증거

- 실행일: 2026-07-15
- 결과: PASS
- 명령: `cargo xtask load100`
- 장비: Apple M4 Pro, 48 GiB RAM, macOS 26.5.2 arm64
- 런타임: Rust 1.96.0, libvips 8.18.0, FFmpeg 8.1.2

## 부하 조건

- FFmpeg로 생성한 실제 4000×3000 JPEG 1개를 100개 독립 upload/job으로 처리
- fixture SHA-256: `386e332da498f3617095a3fd6615dde4756a969050fdc509564a623667fb5ded`
- worker 동시성 4, libvips native thread 1/job
- 실제 `g7mb-sandbox` subprocess로 100회 probe와 100회 JPEG thumbnail 생성
- 최초 작업 10개는 worker crash를 모사해 lease를 반납하지 않고, 2초 만료 후 다른 worker가 재선점
- SQLite WAL, 원본 bounded file download, SHA-256, derivative publish 전체 worker 경로 사용
- RSS는 test runner와 모든 descendant sandbox process 합계를 100ms 간격으로 측정
- 상한: process tree peak RSS 1,572,864 KiB, wall time 240초

동시 worker가 derivative transaction의 read snapshot을 writer로 승격할 때 발생할 수 있는
`SQLITE_BUSY_SNAPSHOT`을 제거하기 위해 해당 경로는 `BEGIN IMMEDIATE`로 writer를 먼저
예약합니다. 수정 후 추가 transient retry 없이 100개 작업을 완료했습니다.

## 증거 경계

이 게이트는 실제 libvips 변환, SQLite queue, subprocess RSS, lease crash recovery를
검증합니다. 원본 object store는 동일 fixture를 bounded local file로 제공하는 credential-free
하네스이므로 실제 AWS S3/R2 네트워크 처리량과 5GiB multipart 증거를 대신하지 않습니다.
cgroup과 25,000px heavy-image는 각각 별도 증거 문서로 검증했습니다. 로컬 Homebrew의 HEIF loader ABI 불일치도 별도
차단선으로 남아 있습니다.
